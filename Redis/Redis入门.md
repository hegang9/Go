## 适用场景

缓存系统：最常用场景，减轻数据库压力

会话存储：用户会话、登录信息

排行榜/计数器：利用有序集合实现实时排行

消息队列：使用 List 或 Stream 实现

实时系统：如实时分析、实时消息推送

分布式锁：实现跨进程/跨服务互斥

地理位置应用：Geo 数据类型支持位置计算

## 单线程模型
Redis 6.0 之前，核心网络请求处理和数据操作是单线程的

为什么采用单线程：
1. 避免多线程的竞争条件和锁开销
2. 充分利用内存速度和CPU缓存
3. 瓶颈通常在网络I/O而非CPU，使用多线程占用CPU更多资源来加速效果甚微 

## I/O多路复用
用于高效处理大量客户端连接

工作原理：
客户端连接 → I/O多路复用器（select/epoll/kqueue） → 事件队列 → 单线程顺序处理

实现方式：epoll

## 为什么高性能
1. 纯内存操作：支持纳秒级访问
2. 单线程模型：无锁竞争，顺序操作
3. I/O多路复用：高效处理网络请求
4. 高效数据结构：专门优化的底层数据结构
5. 合理的持久化策略：RDB快照 + AOF日志

## Key设计规范
1. 命名规范：推荐使用 业务:子模块:唯一标识
2. 设计原则：
   - 长度控制：最好不超过 44 字节，因为redis对短key有优化
   - 避免特殊字符：空格、换行、引号、
   - 包含版本
3. 大 Key/热 Key预防
    - 大 Key：单个 value 超过 10KB 或元素过多（如使用Hash/List作为Key）
    - 热 Key：QPS超过 10w 的 key
    - 解决方案：拆分、本地缓存、读写分离
