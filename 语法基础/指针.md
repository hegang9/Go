# Go语言指针 (Pointer)

在 Go 语言中，指针是一种专门用来存储**内存地址**的变量。通过指针，我们可以直接读取或修改该内存地址上存储的值。

与 C/C++ 相比，Go 语言中的指针被精简了，**不支持复杂的指针算术运算**（如 `p++`），这使得 Go 语言的指针更加简单、安全。

## 1. 核心操作符：`&` 和 `*`

Go 语言中的指针操作主要涉及两个核心操作符：

- **`&` (取地址符)**：放在变量前面，用于获取该变量的内存地址。
- **`*` (解引用符 / 取值符)**：
  1. 放在类型前面（如 `*int`），表示这是一个指向该类型的指针变量。
  2. 放在指针变量前面（如 `*p`），表示获取该指针所指向的具体值，或者对该地址存放的值进行修改。

### 基本操作示例
```go
package main

import "fmt"

func main() {
    a := 10       // 定义一个普通变量 a
    p := &a       // 获取变量 a 的内存地址，赋值给指针 p。此时 p 的类型推导为 *int

    fmt.Println(a)    // 输出: 10
    fmt.Println(p)    // 输出类似于: 0xc00001a070 (运行时分配的内存地址)
    fmt.Println(*p)   // 输出: 10 (通过指针解引用获取该地址中的值)

    *p = 20           // 通过指针修改该内存地址上的值
    fmt.Println(a)    // 输出: 20 (a 的值也随之同步改变)
}
```

## 2. 指针的声明与初始化

### 2.1 声明指针变量
```go
var p *int      // 声明一个指向 int 类型的指针
var p2 *string  // 声明一个指向 string 类型的指针
```
**注意**：在 Go 语言中，指针未初始化时的默认零值是 `nil`（空指针）。如果直接对 `nil` 指针进行取值或赋值（例如直接执行 `*p = 10`），会导致程序崩溃抛出 panic 错误。

### 2.2 使用 `new` 函数初始化
内置函数 `new` 专门用于为某种类型分配内存。它会返回该类型的指针，并将该内存区域初始化为对应类型的**零值**（比如 int 为 0，string 为 ""）。
```go
p := new(int)   // 开辟一块内存用来存放 int 并赋零值 0，返回它的首地址
fmt.Println(*p) // 输出: 0
*p = 100        // 地址和内存已分配，现在赋值是安全的
fmt.Println(*p) // 输出: 100
```

## 3. 为什么需要指针？(值传递与引用)

Go 语言中的函数参数传递**全部都是值传递**。如果在函数中修改传入的参数，实际上修改的只是参数的局部副本，不会影响外部的原变量。

如果你希望**在函数内部修改外部的变量**，或者希望能**避免传递过于庞大的结构体（数据对象）时发生的内存大范围拷贝开销**，就需要使用指针。

### 示例：不使用指针（无法修改原变量）
```go
func modifyValue(x int) {
    x = 100 // 这里只是修改了 x 在当前函数栈中的局部拷贝
}

func main() {
    a := 10
    modifyValue(a)
    fmt.Println(a) // 依然输出: 10 (外部变量 a 未受影响)
}
```

### 示例：使用指针（成功修改原变量）
```go
func modifyPointer(x *int) {
    *x = 100 // 顺藤摸瓜，直接修改了外部指针所指向的那块真实内存上的值
}

func main() {
    a := 10
    // 注意：我们要传入 a 的内存地址
    modifyPointer(&a)
    fmt.Println(a) // 输出: 100 (成功修改了外部的变量)
}
```

## 4. 指针的限制：无指针运算

在 C 和 C++ 等语言中，你可以通过给指针加上偏移量（例如 `p++`、`p = p + 4`）来让指针移动，从而访问数组或内存空间中相邻的其他内存地址。虽然高度灵活，但也极易引发内存越界、野指针等严重的安全漏洞。

**Go 语言从语法层面严格禁止了常规的指针算术运算**。如果你尝试执行 `p++`，编译器直接不予通过。如果你在某些极端追求极致性能或直接与 C 代码等底层交互的场景下，确实需要进行底层的指针运算，需要借助标准库的 `unsafe` 包中的 `unsafe.Pointer`（非常规操作，普通业务开发不推荐使用）。
