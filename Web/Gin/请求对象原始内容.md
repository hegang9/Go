## Gin 中与请求对象原始内容交互

在 Gin 里，绝大多数日常参数处理都可以通过 `c.Param()`、`c.Query()`、`ShouldBindJSON()` 等高级 API 完成。  
但在一些场景下，你需要直接和 **HTTP 请求对象原始内容** 打交道，例如：

- 网关/中间件要记录原始请求体（审计、签名验签）
- 接入第三方回调，数据格式不固定
- 处理大文件流、二进制协议、Webhook 原文校验
- 需要访问 `net/http` 的底层字段能力

Gin 的核心点是：`*gin.Context` 内部持有原生请求对象 `*http.Request`，即 `c.Request`。  
所以你可以在“Gin 便利能力”和“net/http 原始能力”之间无缝切换。

---

### 1. 原始请求对象入口：`c.Request`

`c.Request` 的类型是 `*http.Request`，可以直接访问标准库字段：

- `c.Request.Method`：请求方法（GET/POST/...）
- `c.Request.URL`：完整 URL 对象（含 Path、RawQuery）
- `c.Request.Header`：请求头（`http.Header`）
- `c.Request.Body`：原始请求体流（`io.ReadCloser`）
- `c.Request.Host`、`RemoteAddr`、`TLS`：连接层信息
- `c.Request.Context()`：标准 `context.Context`（取消、超时、链路值）

示例：

```go
r.Any("/raw", func(c *gin.Context) {
	req := c.Request

	c.JSON(200, gin.H{
		"method":      req.Method,
		"path":        req.URL.Path,
		"raw_query":   req.URL.RawQuery,
		"host":        req.Host,
		"remote_addr": req.RemoteAddr,
		"user_agent":  req.UserAgent(),
	})
})
```

### 2. 读取原始 Body（字节流）

#### 2.1 一次性读取

```go
r.POST("/raw-body", func(c *gin.Context) {
	bodyBytes, err := io.ReadAll(c.Request.Body)
	if err != nil {
		c.JSON(400, gin.H{"msg": "读取请求体失败"})
		return
	}

	c.JSON(200, gin.H{
		"size": len(bodyBytes),
		"body": string(bodyBytes),
	})
})
```

这适合小体积 JSON/Text 等请求体。

#### 2.2 关键坑：Body 是“流”，默认只能读一次

当你 `io.ReadAll(c.Request.Body)` 读完后，后续 `ShouldBindJSON` 通常会拿不到内容。  
常见解决方案有两种：

1. **优先使用 Gin 缓存能力**：`ShouldBindBodyWith`
2. **手动回填 Body**：`io.NopCloser(bytes.NewBuffer(bodyBytes))`

手动回填示例：

```go
r.POST("/verify", func(c *gin.Context) {
	raw, err := io.ReadAll(c.Request.Body)
	if err != nil {
		c.JSON(400, gin.H{"msg": "读取失败"})
		return
	}

	// 这里可做签名验签、日志记录
	log.Printf("raw body: %s", string(raw))

	// 回填，供后续绑定再次读取
	c.Request.Body = io.NopCloser(bytes.NewBuffer(raw))

	var req struct {
		Name string `json:"name"`
	}
	if err := c.ShouldBindJSON(&req); err != nil {
		c.JSON(400, gin.H{"msg": "JSON 解析失败"})
		return
	}

	c.JSON(200, gin.H{"name": req.Name})
})
```

### 3. 读取原始 Query 字符串

当你需要拿“未经框架拆分的原始查询串”用于签名或回放：

```go
r.GET("/raw-query", func(c *gin.Context) {
	rawQuery := c.Request.URL.RawQuery
	parsed := c.Request.URL.Query() // url.Values

	c.JSON(200, gin.H{
		"raw_query": rawQuery,
		"parsed":    parsed,
	})
})
```

- `RawQuery`：原始字符串，可能保留参数顺序信息
- `Query()`：解析后的键值结构，使用更方便

---

### 4. 读取原始 Header

```go
r.GET("/raw-header", func(c *gin.Context) {
	headers := c.Request.Header // map[string][]string

	c.JSON(200, gin.H{
		"authorization": c.Request.Header.Get("Authorization"),
		"all_headers":   headers,
	})
})
```

说明：

- Header 在 Go 中是 `map[string][]string`（一个键可多个值）
- 用 `Get()` 取单值最便捷；要完整值列表可直接读 map

### 5. 原始表单解析（`net/http` 方式）

虽然 Gin 提供 `PostForm`，但你也可以直接用标准库解析：

```go
r.POST("/raw-form", func(c *gin.Context) {
	if err := c.Request.ParseForm(); err != nil {
		c.JSON(400, gin.H{"msg": "ParseForm 失败"})
		return
	}

	// ParseForm 后：URL Query + Body 表单会合并到 Form
	c.JSON(200, gin.H{
		"form":      c.Request.Form,
		"post_form": c.Request.PostForm,
	})
})
```

差异：

- `Form`：合并后的参数（Query + Body）
- `PostForm`：仅 Body 表单参数

### 6. Multipart 原始交互（文件流）

对于大文件上传，通常不要一次性 `ReadAll` 到内存。  
更稳妥方式是让标准库按 multipart 解析并使用临时文件：

```go
r.POST("/raw-multipart", func(c *gin.Context) {
	// 32MB 内存阈值，超出部分落磁盘临时文件
	if err := c.Request.ParseMultipartForm(32 << 20); err != nil {
		c.JSON(400, gin.H{"msg": "ParseMultipartForm 失败"})
		return
	}

	form := c.Request.MultipartForm
	files := form.File["file"]

	c.JSON(200, gin.H{
		"file_count": len(files),
		"values":     form.Value,
	})
})
```

### 7. 请求上下文：`c.Request.Context()`

Gin 可与标准 Context 协同，常用于：

- 请求超时/取消感知
- 向下游（DB、HTTP Client、RPC）传递链路上下文

```go
r.GET("/ctx", func(c *gin.Context) {
	ctx := c.Request.Context()

	select {
	case <-ctx.Done():
		c.JSON(499, gin.H{"msg": "客户端已取消"})
		return
	default:
	}

	c.JSON(200, gin.H{"msg": "context 正常"})
})
```

> 说明：`499` 是很多网关常用的“客户端关闭请求”语义码，非标准码，可按团队规范调整。

### 8. 原始请求抓取中间件（实战模板）

下面这个中间件演示了“读取原始 Body + 回填 + 继续后续处理”的标准姿势：

```go
func RawBodyLogger() gin.HandlerFunc {
	return func(c *gin.Context) {
		if c.Request.Body != nil {
			raw, err := io.ReadAll(c.Request.Body)
			if err == nil {
				// 可以打日志、审计、验签
				log.Printf("[%s] %s raw=%s", c.Request.Method, c.Request.URL.Path, string(raw))
				c.Request.Body = io.NopCloser(bytes.NewBuffer(raw))
			}
		}
		c.Next()
	}
}
```

注册：

```go
r := gin.Default()
r.Use(RawBodyLogger())
```

### 9. 推荐实践与性能注意点

1. **非必要不读原始流**：优先用 Gin 的高层 API，减少样板代码。
2. **读取原始 Body 要有大小限制**：避免被超大请求打爆内存。
3. **日志不要直接落敏感数据**：如密码、令牌、身份证号要脱敏。
4. **需要二次消费 Body 时，必须回填或用 `ShouldBindBodyWith`**。
5. **大文件场景走流式/临时文件方案**，避免 `ReadAll`。

可以在网关层或反向代理层设置最大请求体，例如通过 `http.MaxBytesReader`（原生 `net/http`）限制 Body 大小。

---

### 10. 小结

在 Gin 中与请求对象原始内容交互，本质是掌握这条链路：

- 入口：`c.Request`
- 原始流：`c.Request.Body`（一次性）
- 回填复用：`io.NopCloser(bytes.NewBuffer(raw))`
- 结构化解析：`ShouldBind*`
- 上下文协同：`c.Request.Context()`

理解这套机制后，你就可以在“工程化开发效率”和“底层可控性”之间自由切换。
