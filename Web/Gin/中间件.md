## Gin中的中间件
Gin中的中间件是指在处理请求之前或之后执行的一段代码，可以用于实现一些通用的功能，如日志记录、身份验证、错误处理等。Gin提供了一个非常灵活的中间件机制，可以让开发者轻松地添加和管理中间件。中间件在路由引擎中使用：

## Gin中两种创建路由引擎的方式
### 1. gin.Default()
创建一个已附加Logger和Recovery中间件的路由引擎。Logger中间件会记录每个请求的详细信息并写入标准输出，而Recovery中间件会在发生 panic 时恢复并返回500错误，防止服务器崩溃。

## 使用中间件
Gin支持三个级别的中间件附加：

- 全局中间件：应用于路由器中的每个路由，使用`router.Use()`方法添加。
- 分组中间件：应用于路由组中的所有路由，使用`group.Use()`方法添加。
- 路由级中间件：仅应用于单个路由，作为**额外参数**传递给`route.GET()`和`router.POST()`等方法。
```go
// 1. Global -- applies to all routes
router := gin.New()
router.Use(Logger(), Recovery())

// 2. Group -- applies to all routes in the group
v1 := router.Group("/v1")
v1.Use(AuthRequired())
{
  v1.GET("/users", listUsers)
}

// 3. Per-route -- applies to a single route
router.GET("/benchmark", BenchmarkMiddleware(), benchHandler)
```

## 中间件执行顺序
在 Gin 框架中，中间件与处理函数的执行顺序呈现一种类似洋葱模型或**栈模式（先进后出）**的结构。具体执行顺序如下：

- 前置阶段（按注册顺序由外到内执行）：
请求到达时，中间件会严格按照注册的顺序依次执行它们在 c.Next() 之前的代码。
- 核心处理阶段：
当所有中间件的前置阶段执行完毕后，执行最终的路由处理函数（Handler）。
- 后置阶段（按注册顺序反向由内到外执行）：
处理函数执行完成后，程序控制权会从最内层逐级返回，依次执行各个中间件 c.Next() 之后的代码。第一个注册的中间件最先开始执行，但它的后置阶段最后结束。

## 自定义中间件
开发者可以通过编写一个函数来创建自定义中间件，该函数接受一个`*gin.Context`参数并返回一个`gin.HandlerFunc`类型的函数。

中间件函数分为两个阶段，由`c.Next()`分隔：
- 前置处理阶段：在调用`c.Next()`之前执行，可以用于请求预处理，如验证、日志记录或使用c.Set()设置上下文值等。
- `c.next`：调用链中的下一个处理函数，可以是下一个中间件或最终的处理函数。
- 后置处理阶段：在`c.Next()`返回即主处理函数完成后执行，可以用于响应后处理，如清理资源、记录响应日志或根据上下文值进行额外操作等。

## 终止中间件调用链
如果在中间件函数中调用`c.Abort()`，则会终止当前请求的中间件调用链，后续的中间件和处理函数将不会被执行。这通常用于在某些条件下提前返回响应，例如验证失败或权限不足等情况。可与响应结合使用，如`c.AbortWithStatusJSON()`，在终止调用链的同时返回一个JSON响应。

## 错误处理中间件
在典型的 RESTful 应用中，你可能会在任何路由中遇到错误——无效输入、数据库故障、未授权访问或内部 bug。在每个处理函数中单独处理错误会导致重复代码和不一致的响应。

集中式的错误处理中间件通过在每个请求后运行并检查通过 c.Error(err) 添加到 Gin 上下文中的任何错误来解决这个问题。如果发现错误，它会发送一个带有正确状态码的结构化 JSON 响应。
```go
func ErrorHandler() gin.HandlerFunc {
  return func(c *gin.Context) {
    c.Next() // Process the request first

    // Check if any errors were added to the context
    if len(c.Errors) > 0 {
      err := c.Errors.Last().Err

      c.JSON(http.StatusInternalServerError, gin.H{
        "success": false,
        "message": err.Error(),
      })
    }
  }
}
```

## 内置BasicAuth中间件
Gin提供了一个内置的BasicAuth中间件，可以用于实现基本的HTTP身份验证。它接受一个`gin.Accounts`类型的参数，该类型是一个`map[string]string`，表示用户名和密码的键值对，并保护应用它的任何路由组

注意：HTTP 基本认证以 Base64 编码字符串传输凭据，未加密。任何能拦截流量的人都可以轻易解码凭据。在生产环境中使用 BasicAuth 时务必使用 HTTPS（TLS）。

## 中间件中的Goroutine
**在中间件或处理函数中启动新的 Goroutine 时，不应该在其中使用原始上下文，必须使用只读副本。**因为Gin使用sync.Pool 来复用 gin.Context 对象以提高性能，如果在新的 Goroutine 中使用原始上下文，可能会导致数据竞争和不可预测的行为。正确的做法是使用`c.Copy()`方法创建一个只读副本，并在新的 Goroutine 中使用这个副本。

## 依赖注入模式
随着 Gin 应用的增长，你需要一种简洁的方式在处理函数之间共享数据库连接、配置和服务等依赖。Go 的简洁性鼓励使用直接的模式，而不是重量级的 DI 框架。在 Gin 中，常见的依赖注入模式有以下三种：

### 1. 闭包模式
最符合 Go 习惯的方式是使用闭包来传递依赖。这种模式适用于中小型应用，每个处理函数都需要显式声明其依赖。

```go
func GetUserHandler(db *sql.DB) gin.HandlerFunc {
    return func(c *gin.Context) {
        // 使用 db 进行操作
        c.JSON(http.StatusOK, gin.H{"status": "ok"})
    }
}

// 注册路由
r.GET("/users", GetUserHandler(db))
```
优点：编译时类型安全，容易进行 mocks 测试，适合依赖较少的场景。

### 2. 基于结构体的处理函数
对于依赖较多的大型应用，将依赖作为结构体的字段，并将处理函数作为对应结构体的方法，这种方式可以更好地分组和共享依赖。

```go
type App struct {
    DB     *sql.DB
    Logger *slog.Logger
}

func (a *App) GetUser(c *gin.Context) {
    // 使用 a.DB 和 a.Logger 进行操作
    c.JSON(http.StatusOK, gin.H{"status": "ok"})
}

// 注册路由
app := &App{DB: db, Logger: logger}
r.GET("/users", app.GetUser)
```
优点：同样具有编译时类型安全性，容易 mocks 测试，适合中大型项目。

### 3. 中间件注入
使用中间件将依赖注入到 `gin.Context` 请求上下文中。当许多处理函数由于横切关注点需要相同的公共依赖时很有用。

```go
func DatabaseMiddleware(db *sql.DB) gin.HandlerFunc {
    return func(c *gin.Context) {
        c.Set("db", db)
        c.Next()
    }
}

func GetUser(c *gin.Context) {
    db := c.MustGet("db").(*sql.DB)
    // 使用 db
}

// 注册路由
r.Use(DatabaseMiddleware(db))
r.GET("/users", GetUser)
```
缺点与注意事项：由于这种模式使用了 `interface{}` 和类型断言（如 `c.MustGet`），失去了编译时的类型安全性，出错时只能在运行时发现。因此官方推荐**尽可能优先使用闭包或结构体模式**。

### 模式比较总结：
- **闭包**：提供编译时类型安全，容易 Mock，适用于小型应用、少量依赖。
- **结构体**：提供编译时类型安全，容易 Mock，推荐用于中大型应用中的依赖共享。
- **中间件**：运行时类型安全（使用装箱/拆箱机制），容易共享状态，但会丢失编译期检查。通常用于处理横切关注点（如全局配置、当前登录用户会话等）。
