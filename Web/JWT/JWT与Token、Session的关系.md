# JWT与Token、Session的关系

在Web开发中，HTTP协议是无状态的（Stateless），这意味着服务器默认不知道两次请求是否来自同一个用户。为了解决“记住用户身份”的问题，演进出了 Session、Token 以及 JWT 这几种主要的认证机制。

## 1. Session（会话）
**工作原理**：
1. 用户登录成功后，服务器在内存或数据库（如Redis）中创建并保存一段用户的状态信息（即 Session）。
2. 服务器生成一个唯一的 `Session ID`，通过 HTTP 响应头（`Set-Cookie`）返回给客户端。
3. 客户端浏览器自动将 `Session ID` 保存在 Cookie 中，并在后续请求时自动带上。
4. 服务器收到请求后，通过 Cookie 中的 `Session ID` 去内存或数据库中查找对应的 Session 信息，从而识别用户身份。

**特点**：
- **状态保存在服务端**：服务器需要消耗内存或存储资源来保存所有在线用户的 Session。
- **安全性较高**：核心状态留在服务器，客户端只有一层“钥匙”（Session ID）。
- **缺点**：难以水平扩展（分布式架构下需要做复杂的 Session 共享或粘性会话）；强依赖 Cookie（存在 CSRF 风险，且跨域和移动端原生支持较差）。

## 2. Token（令牌）
**工作原理**：
Token 是一个更广泛的概念，指的是“身份凭证”。它本身并不是某种具体的标准，而是一种通过凭证访问的方式。
1. 用户登录成功后，服务器生成一个随机的、无规则的字符串（Token）作为凭证。
2. 服务器通常会将这个 Token 与用户信息关联，并存入数据库或缓存。
3. 把 Token 下发给客户端，客户端自行保存（如 LocalStorage、SessionStorage 等）。
4. 客户端后续请求时，主动将 Token 放入 HTTP 请求头（通常是 `Authorization: Bearer <token>`）发送给服务器。
5. 服务器收到 Token 后，去数据库/缓存查询比对，验证其有效性并获取用户信息。

**特点**：
- 解决了跨域（CORS）问题，因为不强依赖 Cookie。
- 抵御 CSRF 攻击（由于不再是由浏览器自动携带 Cookie，而是由前端代码主动附加到请求头中）。
- **本质仍然是“有状态的”**：即服务器依然需要保存 Token，并进行查库验证。

## 3. JWT（JSON Web Token）
**工作原理**：
JWT 是 Token 的一种具体实现和标准化格式。与传统的 Opaque Token（随机无意义字符串）不同，JWT 是**自包含的（Self-contained）**，**它本身就携带了验证所需的用户数据**。
1. 用户登录成功后，服务器根据用户信息拼接成 JSON，然后使用只有服务器知道的密钥进行哈希签名，最终生成一个 JWT（具有 `Header.Payload.Signature` 结构）。
2. 服务器**不需要将 JWT 保存在服务端**（不消耗服务器存储空间），直接将它下发给客户端。
3. 客户端保存 JWT，并在后续请求的 Header 中携带。
4. 服务器收到请求后，不去查数据库，而是**使用本地存放的密钥对 JWT 的签名部分进行计算校验**。如果计算出的签名和 JWT 自带的签名一致，并且有效期（exp）也没过，就信任并直接从 Payload 中读取用户信息。

**特点**：
- **完全无状态（Stateless）**：服务器不需要保存任何会话信息，也不需要查数据库就能知道用户的身份，极大地方便了分布式和微服务架构的横向扩展。
- **缺点**：
  - **难以主动撤销**：因为服务器不保存状态，一旦 JWT 签发，在它自身的超时时间到达前，服务器很难主动让它失效（例如用于踢人下线，或者改密后失效旧令牌。通常得借助额外的黑名单设计，但这又变回了有状态）。
  - **带宽高消耗**：JWT 包含了明文信息和长签名，长度远超简单的 Session ID 字符串。
  - **数据可见性**：Payload 只是 Base64 编码，并未加密，任何人都可以解码看到明文内容，因此绝对不能在其中存放密码等敏感信息。

## 总结与关系比喻
- **Session**：服务端存了一份档案说“你是合法用户”。只发给你一把纯数字钥匙（Session ID），你每次拿钥匙来开门，我都要去查验这把钥匙对应的档案是谁的。
- **Token**：一个更泛化的“访问票据”概念。它一般由客户端保管并在请求头中携带。传统的 Token 依然需要在服务端进行比对查询（相当于需要去查真伪的票据）。
- **JWT**：Token 的一种具体进阶实现，属于**无状态 Token**。它不仅是一张票据，还在票面直接写明了“你的名字、权限、有效期”，并且在票据上盖了个密码学防伪公章（哈希签名）。服务端只要看到公章是真的，就直接信任票据上的内容，**省去了去档案库查档案的步骤**。
