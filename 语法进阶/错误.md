## 异常类别
Go中有三种异常类别：
1. error：流程出错，需要处理，但不处理程序也不会崩溃
2. panic：很严重的问题，程序无法继续执行了，在处理完问题后（继续执行defer语句）后崩溃
3. fatal：非常严重的问题，程序无法继续执行了，必须立刻崩溃，并且不运行任何defer语句

在大多数情况下，Go中的异常会作为函数的返回值进行判断和处理。这样心智负担小，易于调试；但是错误没有堆栈信息，且自定义错误是变量而非常量，会导致变量遮蔽问题

## error
error本身是一个预定义的接口类型，该接口只有一个方法：`Error() string`，它的作用是**返回一个描述错误的字符串**。Go语言中，函数如果可能会发生错误，通常会返回一个error类型的值来表示错误情况。

```go
type error interface{
    Error() string
}
```

### 创建error
1. 使用`errors`包中的`New`函数创建一个简单的error对象
```go
import "errors"

err := errors.New("something went wrong")
```

2. 使用`fmt`包中的`Errorf`函数创建一个格式化参数的error对象
```go
import "fmt"

err := fmt.Errorf("failed to open file: %s", filename)
```

为了代码的可维护性，一般不会创建临时error，而是会将常用的error当作**全局变量**使用

```go
var ErrNotFound = errors.New("not found")
```

### 自定义error
通过实现`Error()`方法可以自定义`error`
```go
func New(text string) error {
   return &errorString{text}
}

// errorString结构体
type errorString struct {
   s string
}

func (e *errorString) Error() string {
   return e.s
}
```

### 传递
在 Go 里，错误传递的本质是：

1. 当前函数发现错误。
2. 当前函数决定是否在本层处理。
3. 如果不能或不应该处理，就把 `error` 作为返回值**上抛给调用方**。

这也是 Go 推荐的错误处理模型：**谁最了解业务语义，谁处理错误；否则就继续传递。**

### 传递的基本写法

```go
func readConfig(path string) ([]byte, error) {
   data, err := os.ReadFile(path)
   if err != nil {
      return nil, err // 原样上抛
   }
   return data, nil
}
```

上面的 `return nil, err` 就是最基础的错误传递：本层不吞掉错误，交给上层决定。

### 传递时补充上下文（推荐）

真实项目中，单纯返回原始错误往往信息不足。推荐用 `fmt.Errorf` + `%w` 包裹错误，补充“在哪一步失败”：

```go
func loadUserConfig(path string) ([]byte, error) {
   data, err := os.ReadFile(path)
   if err != nil {
      return nil, fmt.Errorf("读取用户配置失败(path=%s): %w", path, err)
   }
   return data, nil
}
```

这里 `%w` 的作用是“保留原始错误”，形成错误链（error chain）。

### 错误链：errors.Is 与 errors.As

当错误被层层包裹后，不能再用 `==` 直接比较，应使用 `errors.Is` 和 `errors.As`。

#### 1) 用 `errors.Is` 判断“是不是某个哨兵错误”

```go
var ErrUserNotFound = errors.New("user not found")

func findUser(name string) error {
   return fmt.Errorf("查询用户失败(name=%s): %w", name, ErrUserNotFound)
}

func main() {
   err := findUser("alice")
   if errors.Is(err, ErrUserNotFound) {
      fmt.Println("用户不存在，走兜底逻辑")
      return
   }
   if err != nil {
      fmt.Println("其他错误:", err)
   }
}
```

#### 2) 用 `errors.As` 提取“具体错误类型”

```go
type ValidationError struct {
   Field string
   Msg   string
}

func (e *ValidationError) Error() string {
   return fmt.Sprintf("字段 %s 校验失败: %s", e.Field, e.Msg)
}

func validateAge(age int) error {
   if age < 0 {
      // 为什么这里返回指针 &ValidationError？
      // 因为上面的 Error() 方法是挂载在指针类型 *ValidationError 上的。
      // 只有指针类型才实现了 error 接口。
      return &ValidationError{Field: "age", Msg: "不能小于0"}
   }
   return nil
}

func saveUser(age int) error {
   if err := validateAge(age); err != nil {
      return fmt.Errorf("保存用户失败: %w", err)
   }
   return nil
}

func main() {
   err := saveUser(-1)
   var ve *ValidationError
   if errors.As(err, &ve) {
      fmt.Println("可定位到字段错误:", ve.Field)
   }
}
```

### 一条完整的错误传递链（建议掌握）

```go
func repositoryGet(id int) (string, error) {
   if id == 0 {
      return "", ErrNotFound
   }
   return "Tom", nil
}

func serviceGetName(id int) (string, error) {
   name, err := repositoryGet(id)
   if err != nil {
      return "", fmt.Errorf("service 查询用户失败(id=%d): %w", id, err)
   }
   return name, nil
}

func handler(id int) {
   name, err := serviceGetName(id)
   if err != nil {
      if errors.Is(err, ErrNotFound) {
         fmt.Println("返回 404")
         return
      }
      fmt.Println("返回 500:", err)
      return
   }
   fmt.Println("返回 200, name=", name)
}
```

这一套链路非常典型：

- 底层产出原始错误（如 `ErrNotFound`）。
- 中间层补业务上下文并继续上抛（`%w`）。
- 最上层统一分类处理（404 / 500）。

### 什么时候“处理”而不是“继续传递”

一般在以下场景直接处理：

- **可以恢复**：比如重试、降级、使用默认值。
- **需要转译**：把底层错误映射成业务错误码。
- **边界层输出**：HTTP Handler、CLI main、RPC 网关等负责最终响应或日志。

否则就应继续传递，避免“吞错”（把错误吃掉却不返回）。

### 常见错误传递陷阱

1. **丢失上下文**
   只 `return err`，却不知道在哪一步失败。应在关键层用 `%w` 增加上下文。

2. **错误比较方式错误**
   包裹后继续用 `==` 比较，可能失效。应改用 `errors.Is`。

3. **该包裹却没包裹 / 不该包裹却乱包裹**
   - 需要保留原始错误语义时，用 `%w`。
   - 只是打印信息不参与错误链判断时，可用 `%v`。

4. **忽略错误**
   出现 `_ = err` 或者直接不判断 `err`，会让问题在后面放大。

### 实战建议（简洁版）

- 函数签名优先采用 `(..., error)`。
- 每次调用后立刻判断 `if err != nil`。
- 重要边界加上下文：`fmt.Errorf("xxx: %w", err)`。
- 判断类别用 `errors.Is`，提取类型用 `errors.As`。
- 最外层统一收口（日志、返回码、退出码）。

掌握这套“发现 -> 包裹 -> 上抛 -> 分类处理”的链路，你就掌握了 Go 错误传递的核心。

## panic
panic 是 Go 中运行时异常的表达形式，通常在一些危险操作中会出现，主要是为了及时止损，防止造成更加严重的后果。panic 在让程序崩溃之前会做好程序的善后工作，比如执行所有已经注册的 defer 语句，释放资源等，同时 panic 也可以被恢复来保证程序继续运行

注意：当程序中存在多个协程时，只要任一协程发生panic，如果不将其捕获的话，整个程序都会崩溃

### 1. 触发 panic

`panic` 可以由 Go 运行时（Runtime）自动触发，也可以在代码中手动触发。

**手动触发：**
调用内置的 `panic()` 函数，可以传入任意类型的值（通常是字符串或 error）。
```go
func doSomething() {
    // ... 发生严重逻辑错误
    panic("database connection failed!") // 直接让程序崩溃并打印堆栈信息
}
```

**运行时自动触发：**
```go
func makeMistake() {
    var a []int
    a[0] = 1 // 运行时 panic: index out of range
    
    var m map[string]int
    m["key"] = 1 // 运行时 panic: assignment to entry in nil map
}
```

### 2. 捕获 panic (recover)

虽然 `panic` 会导致程序崩溃，但 Go 提供了内置函数 `recover()` 来“拦截”这种崩溃，使程序恢复控制权，也就是常常说的“善后处理”。

**极其重要：** `recover()` 必须且只能在 `defer` 语句声明的闭包函数内部调用才有效！如果在正常执行流程里调用 `recover()`，它只会返回 `nil`，什么也做不了。

```go
func safeCall() {
    defer func() {
        // 发生 panic 时，函数的控制流会立刻终止并跳转执行所有的 defer
        if err := recover(); err != nil {
            fmt.Println("成功捕获到 panic:", err)
            // 在这里可以记录日志、报警，或者将 panic 转化为普通的 error 返回给外面
        }
    }()

    fmt.Println("开始执行...")
    panic("发生了一个严重错误！")
    
    // panic 后面的代码永远不会被执行
    fmt.Println("这行打印不出来")
}

func main() {
    safeCall()
    fmt.Println("程序没有崩溃，继续运行 main 函数！")
}
```

### 3. 注意事项与常见陷阱

#### 1) 协程边界隔离（Goroutine 限制）
**`recover()` 只能捕获当前 Goroutine（协程）中发生的 `panic`。** 
如果你在主协程中写了 `defer recover()`，但由它派生出的子协程中发生了 `panic`，主协程是**捕获不到**的，整个程序依然会崩溃！

```go
func main() {
    defer func() {
        if err := recover(); err != nil {
            fmt.Println("主协程尝试捕获 panic:", err)
        }
    }()

    go func() {
        panic("子协程发生了 panic！") // 程序会直接崩溃，主协程的 recover 无效
    }()

    time.Sleep(time.Second)
}
```
**正确做法：** 必须在每个可能发生 `panic` 的新协程内部，写上自己的 `defer recover()`：
```go
go func() {
    defer func() {
        if err := recover(); err != nil {
            // 只捕获自己协程里的 panic
        }
    }()
    panic("子协程发生了 panic！")
}()
```

#### 2) 千万不要用 panic 替代 error 作流程控制
Go 并不是 Java、Python 或 C++。在 Go 中，**`panic` 和 `recover` 绝不能被当作 try/catch 来进行常规的错误处理或控制流扭转**。
普通的业务错误（比如“找不到用户”、“密码错误”、“参数不合格”）必须且只能通过返回 `error` 来进行传递和处理！滥用 `panic` 会使代码逻辑混乱，且每次捕捉 `panic` 都会带来明显的性能开销。

#### 3) 什么时候真正应该用 panic？
*   **不可恢复的灾难性错误（启动阶段）**：比如服务进程刚启动时，必须读取配置或者建立数据库连接。如果失败了，程序勉强启动也没有意义，此时应该直接 `panic` 退出。
*   **开发者层面不可原谅的 Bug**：比如调用了一个明确约定某参数切片长度不能为 0 的内部核心算法函数，或者某个不该到达的 `switch default` 分支。这代表代码里有严重的逻辑死角。
*   **初始化阶段的便捷函数**：Go 标准库里有很多以 `Must` 开头的函数，比如正则表达式包里的 `regexp.MustCompile`。这代表如果正则字符串写错了（一般是常量），就直接引发 `panic` 暴露在编译测试期，省去了每次启动都要写 `if err != nil` 的繁琐。

## fatal

在 Go 语言本身的关键字或内建语法中，其实并没有一个专门叫 `fatal` 的错误类型。当我们谈论 Go 中的 `fatal`（致命错误）时，通常是指**直接调用 `os.Exit()` 或者标准库中的 `log.Fatal`、`log.Fatalf` 等函数**来让程序**立即、无条件地停止运行**。

### 1. 什么是 fatal？

`fatal` 级别的错误代表了程序遇到了**绝对无法继续执行下去、也没必要进行任何挽救**的灾难性状况。

当我们调用 `log.Fatal("服务器启动失败")` 时，它的底层实际上只做了两件事：
1. 打印你传入的错误日志。
2. 隐式地调用 `os.Exit(1)` 退出整个进程，返回非零的状态码（表示异常退出）。

### 2. fatal 与 panic 的核心区别（非常重要！）

很多初学者容易把 `fatal` 和 `panic` 混淆，它们都会让程序崩溃，但两者的行为截然不同：

*   **`panic` 会体面地退出**：它会沿着调用栈一层一层往上退，**会执行所有的 `defer` 函数**（这就意味着它可以关闭已打开的文件、释放已获取的锁等），并且允许被 `recover()` 拦截恢复。
*   **`fatal` 是一剑封喉**：调用 `os.Exit()` 后，程序**立即终止**。**任何已经注册的 `defer` 语句都不会被执行！** 因此，如果你的代码里有非常关键的资源清理工作放在了 `defer` 中，使用 `fatal` 会直接导致这些清理工作被跳过。

**代码对比演示：**

```go
func testFatal() {
    defer fmt.Println("这是一句 defer，但在 fatal 面前它不会被执行！")

    fmt.Println("准备调用 log.Fatal...")
    log.Fatal("发生致命错误，程序立刻终止！") // 👈 直接结束进程，后面的代码和 defer 全被跳过
    
    fmt.Println("这行永远打印不出")
}
```

### 3. 使用场景与最佳实践

因为 `fatal` 过于“暴力”（不执行 defer），我们在日常开发中对它的使用必须极其克制。

#### 什么时候该用（也就是用在 `main` 函数或 `init` 初始化阶段）：
1.  **服务启动阶段必需的资源不可用**：例如你的 Web 服务启动时，发现配置文件读取不到、监听的端口被占用了，或者核心的 MySQL 数据库根本连不上。此时程序毫无继续运行的意义，直接用 `log.Fatal` 或 `log.Fatalf` 报错退出并且抛出异常退出码是最合适的。
2.  **CLI 命令行工具**：在编写命令行脚本（CLI）时，如果用户输入了非法的参数可以直接输出错误并退出。

#### 什么时候永远不要用：
1.  **Web 服务的每次请求处理中（Handler）**：千万不能在处理单个 HTTP 请求或者 RPC 调用的业务逻辑里使用 `log.Fatal`。如果这里使用 `fatal`，某个用户的单个异常请求会直接把整个服务进程杀掉，导致所有其他用户的请求也全部断开！
2.  **在被其他人调用的公共库 (Library) / 包代码中**：作为一个公共工具包，你永远不知道调用者是希望能处理你的错误还是退出。如果你包里的代码私自调用了 `os.Exit` 或 `log.Fatal`，对于调用方来说这是灾难性的。在包和库中，**必须永远返回 `error`，由最上层的调用方（如 `main` 函数）来决定要不要 `fatal`**。
